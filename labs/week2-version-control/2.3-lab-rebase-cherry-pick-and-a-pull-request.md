# 2.3 Lab: Rebase, Cherry-Pick and a Pull Request

> **Week 2 · Lesson 3 · Indicative time: 40 minutes · Supports WLO2, WLO4**
> Part of the CSO7024 DevOps lab series. Companion to the Week 2 module content.

## Overview

In this hands-on lab you practise interactive rebase and cherry-pick in your own repository, then open your first pull request on GitHub. The aim is to give you a complete end-to-end experience of the modern collaborative Git workflow before you meet it under assessment pressure.

## Learning outcomes

By the end of this lab you will be able to:

- Tidy a messy commit history with an interactive rebase.
- Move a single commit between branches with `git cherry-pick`.
- Push a branch to a remote and open a pull request with a clear title, description and reviewable diff.

## Before you start

- Continue working in the `devops-week2` repository from labs 2.1 and 2.2.
- You now need a GitHub account and command-line authentication. If you have not set these up, see [`SETUP.md`](../../SETUP.md), section 3.

## Steps

### Step 1 — Create a remote and push

Create a new **private** repository on GitHub (do not initialise it with a README, since your local repository already has one). Then connect your local repository to it and push:

```bash
git remote add origin <your-repository-url>
git push -u origin main
```

Refresh the repository page on GitHub. Your history should appear.

### Step 2 — Tidy history with an interactive rebase

Create a branch to work on so you never rebase shared history on `main`:

```bash
git switch -c week2-contribution
```

Make three small commits with deliberately untidy messages (for example `wip`, `fix typo`, `more`). Then rewrite them into a clean history:

```bash
git rebase -i HEAD~3
```

In the editor, change some `pick` lines to `squash` (or `s`) to combine commits, and to `reword` (or `r`) to improve messages. Save and exit, then edit the resulting commit message when prompted. Inspect the result:

```bash
git log --oneline
```

You should see a tidier history than before. This is the value of rebase: a readable story of the change rather than a log of every keystroke.

### Step 3 — Cherry-pick a commit

Switch to `main`, make a small commit there (for example a one-line change to `notes.md`), and note its hash with `git log --oneline`. Then bring just that commit onto your contribution branch:

```bash
git switch week2-contribution
git cherry-pick <commit-hash>
```

Confirm the commit now appears on `week2-contribution` as well. Cherry-pick is the right tool when you want one specific commit, not a whole branch's history.

### Step 4 — Push the branch and open a pull request

```bash
git push -u origin week2-contribution
```

On GitHub, open a pull request from `week2-contribution` into `main`. Give it a clear title and a short description that says what the change does and why. Open the **Files changed** tab and read the diff as a reviewer would.

### Step 5 — Review and merge

Because this is your own repository, you can merge the pull request yourself. Review the diff carefully first. In a real team this review-and-approve step is carried out by someone other than the author; that separation is the whole point of the pull request.

### Step 6 — Capture a screenshot

Take a screenshot of the open pull request showing the title, description and diff, and a second screenshot after merging.

## Success criteria

You have completed this lab when your repository on GitHub shows the pushed `week2-contribution` branch and a pull request from that branch into `main`, with a clear title, a short description and a reviewable diff.

## Stretch goal (optional)

Pair with a peer: each of you opens a pull request on the other's repository and leaves a review comment before approving. This lets you experience the review separation that a real team relies on, where the author and the reviewer are different people.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `git push` rejected as non-fast-forward | The remote has commits you do not have locally | `git pull --rebase` then push again |
| Interactive rebase shows more or fewer commits than expected | `HEAD~3` did not match the commits you meant | Quit with `git rebase --abort`, check `git log --oneline`, retry with the right count |
| `git cherry-pick` reports a conflict | The commit does not apply cleanly | Resolve the conflict as in lab 2.2, then `git cherry-pick --continue` |
| Authentication fails on push | Command-line auth not configured | Re-run `gh auth login`, or use an SSH remote (see `SETUP.md`) |
