# 2.2 Lab: Branching, Merging and a Three-Way Conflict

> **Week 2 · Lesson 2 · Indicative time: 40 minutes · Supports WLO2, WLO3**
> Part of the CSO7024 DevOps lab series. Companion to the Week 2 module content.

## Overview

In this hands-on lab you create a feature branch, integrate it back into `main` using both a fast-forward merge and a true (three-way) merge, then deliberately create a conflict so that you can practise resolving it. The aim is to make conflict resolution feel like a routine engineering task rather than an emergency.

## Learning outcomes

By the end of this lab you will be able to:

- Create and switch between branches.
- Distinguish a fast-forward merge from a true three-way merge in the commit graph.
- Read Git's conflict markers and resolve a merge conflict by hand.

## Before you start

- Continue working in the `devops-week2` repository you created in lab 2.1. If you skipped it, complete lab 2.1 first.
- Open a terminal in that repository. Every command below is run there.

## Steps

### Step 1 — Confirm where you are

```bash
git status                         # should be clean, no uncommitted changes
git log --oneline --graph --all    # review the existing history
```

### Step 2 — Fast-forward merge

```bash
git switch -c add-license          # create and switch to a new branch
```

Create a file `LICENSE` containing a single line, `MIT License - practice repository`, then:

```bash
git add LICENSE && git commit -m "Add license file"
git switch main
git merge add-license              # this is a fast-forward; no merge commit is created
git log --oneline --graph --all    # notice the linear history
```

### Step 3 — True three-way merge (no conflict)

```bash
git switch -c add-contributors
```

Create a file `CONTRIBUTORS.md` listing your name, then:

```bash
git add CONTRIBUTORS.md && git commit -m "Add contributors file"
git switch main
```

Edit `notes.md` and add a new line about branching, then:

```bash
git add notes.md && git commit -m "Add note on branching"
git merge add-contributors         # Git creates a merge commit; your editor opens for the message, save and exit
git log --oneline --graph --all    # notice the merge commit and the diamond shape in the graph
```

### Step 4 — Create and resolve a conflict

```bash
git switch -c rewrite-readme
```

Replace the contents of `README.md` with:

```
# DevOps Week 2 - Practice Repository

This repository is used to practise Git fundamentals.
```

Commit it, then make a competing change on `main`:

```bash
git add README.md && git commit -m "Rewrite readme"
git switch main
```

Edit `README.md` on `main` and change the existing first line to:

```
# Devops Week 2 Lab

Lab repo for the module.
```

Commit that, then merge the branch and watch the conflict appear:

```bash
git add README.md && git commit -m "Update readme heading on main"
git merge rewrite-readme           # Git reports a conflict and stops
git status                         # README.md is listed as conflicted
```

Open `README.md` in your editor. You will see conflict markers: `<<<<<<< HEAD`, the version from `main`, `=======`, the version from `rewrite-readme`, then `>>>>>>> rewrite-readme`. Edit the file by hand to produce the version you want to keep, removing all the markers. Then:

```bash
git add README.md                  # mark the conflict as resolved
git commit                         # complete the merge; the editor opens with a default message, save and exit
git log --oneline --graph --all    # observe the merge commit
```

### Step 5 — Capture a screenshot

Take a screenshot of your final `git log --oneline --graph --all` output, showing the fast-forward merge and both true merges in the history.

## Success criteria

You have completed this lab when your final `git log` graph shows a fast-forward merge, a true merge with no conflict and a true merge that resolved a conflict.

## Stretch goal (optional)

During a future conflicting merge, before you have committed, try `git merge --abort` to abandon the merge and return to the pre-merge state. This is the safety net to reach for when a merge is going badly.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `git merge add-license` created a merge commit | `main` had moved on, so it was not a fast-forward | That is fine; the learning point still holds. Reset to redo if you want the linear case |
| Editor will not close after a merge | Your default editor is `vim` and you are unsure how to exit | Type `:wq` then Enter, or set a friendlier editor with `git config --global core.editor nano` |
| Conflict markers remain after committing | The markers were not removed before staging | `git revert` or edit again, remove every `<<<<<<<`, `=======` and `>>>>>>>` line, then re-stage |
