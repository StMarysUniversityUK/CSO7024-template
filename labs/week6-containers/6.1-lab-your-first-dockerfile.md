# 6.1 Lab: Your First Dockerfile

> **Week 6 · Lesson 1 · Indicative time: 45 minutes · Supports WLO2**
> Part of the CSO7024 DevOps lab series. Companion to the Week 6 module content.

## Overview

In this hands-on lab you write your first Dockerfile, build a Docker image for the shared Python application in this repository (the same `app/` you have built on since the Week 3 pipeline), and run that image as a container. By the end you will have produced a portable, runnable artefact that is the foundation for every other activity this week. You will also push the image to a local registry, mirroring the workflow a Continuous Integration pipeline would use.

## Learning outcomes

By the end of this lab you will be able to:

- Write a Dockerfile that produces a small, reproducible image for a Python application.
- Build, tag and run an image, and reach the application running inside the container.
- Explain image layering and observe build-cache reuse between builds.

## Steps

### Step 1 — Start a local registry

A registry is where images are stored and pulled from. Run a throwaway local one:

```bash
docker run -d -p 5000:5000 --name registry registry:2
```

### Step 2 — Write the Dockerfile

In the application directory, create a file named `Dockerfile`. A reference version ships at [`app/Dockerfile`](../../app/Dockerfile); write your own first, then compare. It should:

- Start from a small, pinned base image (`python:3.12-slim`).
- Copy `requirements.txt` and install dependencies **before** copying the application code, so the dependency layer is cached when only code changes.
- Copy the application code.
- Run as a non-root user.
- Expose port 8000 and start the app.

### Step 3 — Build and tag the image

```bash
docker build -t cso7024-app:0.2 .
```

Watch the build output: each instruction in the Dockerfile produces a layer.

### Step 4 — Run the container

```bash
docker run -d -p 8000:8000 --name app cso7024-app:0.2
curl http://localhost:8000/health
```

The health endpoint should return a `200` response with the body `ok`.

### Step 5 — Observe the build cache

Make a one-line change to the application code (not to `requirements.txt`), then rebuild:

```bash
docker build -t cso7024-app:0.2 .
```

Notice that the dependency-installation layer is reused from cache, because only the code changed. This is why copy order in a Dockerfile matters.

### Step 6 — Push to the local registry

```bash
docker tag cso7024-app:0.2 localhost:5000/cso7024-app:0.2
docker push localhost:5000/cso7024-app:0.2
```

This is the same step a CI pipeline performs after a successful build, and it is how the image reaches the Kubernetes lab in Lesson 3.

## Success criteria

You have completed this lab successfully if your image builds without error, the container runs, `curl http://localhost:8000/health` returns a `200` response and you can see the cache reuse on the second build.

## Stretch goal (optional)

Convert the Dockerfile to a multi-stage build that installs dependencies in a builder stage and copies only what is needed into a slim final stage. Add a `HEALTHCHECK` instruction. With these changes the Dockerfile is close to production grade.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `docker build` cannot find `requirements.txt` | The build context does not include it | Run the build from the directory that contains the app, or adjust the `COPY` paths |
| `curl` refuses the connection | The container port is not published, or the app listens on the wrong port | Confirm `-p 8000:8000` and that the app reads `PORT` (default 8000) |
| Push fails with an HTTPS error | Docker is trying to use TLS for the local registry | Use `localhost:5000`, which Docker treats as insecure by default |
| Cache never reused | Code is copied before dependencies | Reorder so `requirements.txt` is copied and installed first |
