# 3.2 Lab: Design and Write Your First CI Pipeline

> **Week 3 · Lesson 2 · Indicative time: 45 minutes · Supports WLO2**
> Part of the CSO7024 DevOps lab series. Companion to the Week 3 module content.

## Overview

In this hands-on lab you design and write a Continuous Integration (CI) pipeline in YAML for the shared Python application in this repository (`app/`). This is the first lab that uses that application; in Week 2 you practised Git on a separate scratch repository. The lab is split into a design phase, where you sketch the pipeline before writing any YAML, and an implementation phase. The pipeline you build here is the pipeline you extend in Lesson 3 and the foundation of the workflow you submit for Assessment 2.

## Learning outcomes

By the end of this lab you will be able to:

- Sketch a pipeline's triggers, stages and dependencies before writing any configuration.
- Write a GitHub Actions workflow that lints, tests and caches dependencies on every push.
- Read a pipeline run in the GitHub Actions interface and interpret a failing job.

## Before you start

- You need a GitHub account and Python 3.12 or later installed. See [`SETUP.md`](../../SETUP.md).
- A starter workflow is provided at [`starter/.github/workflows/ci.yml`](starter/.github/workflows/ci.yml). It is a template with `TODO` markers for you to complete. Note that it will not run from that location. GitHub Actions only runs workflows stored in `.github/workflows/` at the **root** of your repository, so once you have cloned your copy (Step 1) you must move the file there, to `./.github/workflows/ci.yml`. The command is in Step 3.
- The application under test is the shared [`app/`](../../app) at the repository root: `calculator.py` and `tests/test_calculator.py`.

## Steps

### Step 1 — Create your own private copy of the starter

If you are working in your own copy of the module template (recommended), the application and the starter workflow are already present. If your tutor has given you a separate starter repository:

- Open it on GitHub, click **Use this template**, then **Create a new repository**. Give your copy a name, set the visibility to **Private** and create it.
- Do **not** use the **Fork** button: a fork of a public repository is itself public.
- Clone your new repository, then change into the cloned folder.

Confirm the application runs locally before you automate it:

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r app/requirements.txt -r app/requirements-dev.txt
python3 -m pytest app -q
```

### Step 2 — Sketch your pipeline on paper

Before writing any YAML, sketch:

- **The triggers.** Which events should run this pipeline?
- **The stages.** Lint, test, package: are all three needed yet?
- For each stage, what command(s) will run, what dependencies are needed and how long do you expect it to take.
- Any environment variables or secrets the pipeline will need.

### Step 3 — Write the workflow file

First, move the starter workflow to the root of your repository so GitHub Actions will run it. You only do this once:
```bash
mkdir -p .github/workflows
git mv labs/week3-cicd/starter/.github/workflows/ci.yml .github/workflows/ci.yml
```

Then open `./.github/workflows/ci.yml` and complete the `TODO` markers. Your pipeline must:

- Trigger on push to any branch and on pull request to `main`.
- Have a **lint** stage that runs `ruff check .` and fails the pipeline on any error.
- Have a **test** stage that runs `pytest -v` and uploads the test report as an artefact.
- Cache pip dependencies between runs, keyed on a hash of `requirements.txt`.
- Use Python 3.12 on the latest Ubuntu hosted runner.

### Step 4 — Commit, push and observe

```bash
git add .github/workflows/ci.yml
git commit -m "Add initial CI pipeline"
git push origin main
```

Open your repository on GitHub, go to the **Actions** tab and watch your pipeline run. Select each job and read the logs.

### Step 5 — Deliberately break the pipeline and fix it

- Edit `app/calculator.py` and introduce a syntax error, for example a missing colon at the end of a function definition.
- Commit and push, then observe the lint stage failing.
- Read the failure log carefully. Note where the failure is reported, what the error message says and how the status appears in the pull-request interface.
- Fix the syntax error, commit and push again, then watch the pipeline go green.

### Step 6 — Capture a screenshot

Take a screenshot of the successful pipeline run for use in the activity that follows.

## Success criteria

You have completed this lab when your pipeline runs both stages successfully on a push and shows a red status when you introduce a deliberate failure.

## Stretch goal (optional)

Add a job that runs the test suite under both Python 3.11 and Python 3.12 in parallel using a build matrix. Notice how the pipeline visualisation changes.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Workflow does not appear under Actions | File is in the wrong place | It must be at `.github/workflows/ci.yml` in the repository root |
| `ruff: command not found` in the lint job | Dev dependencies not installed in that job | Install `requirements-dev.txt` before running `ruff` |
| Cache never hits | Cache key does not change with the lockfile, or the path is wrong | Key the cache on `hashFiles('**/requirements.txt')` and cache the pip directory |
| Tests pass locally but fail in CI | A path or working-directory difference | Run pytest against the `app` directory in both places |
