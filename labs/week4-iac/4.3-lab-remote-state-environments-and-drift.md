# 4.3 Lab: Remote State, Environments and Drift

> **Week 4 · Lesson 3 · Indicative time: 45 minutes · Supports WLO3**
> Part of the CSO7024 DevOps lab series. Companion to the Week 4 module content.
>
> *Filename note: the Week 4 content cites this brief as "Remote State, Workspaces and Drift". This repository titles it "Environments" because the lab separates environments by directory rather than by Terraform workspaces. See `docs/lab-brief-index.md`.*

## Overview

In this hands-on lab you move the configuration you wrote in lab 4.2 from local state to a remote state backend, separate `dev` and `prod` environments with a directory per environment, deliberately introduce drift and observe Terraform's response. The lab uses a local HTTP backend to simulate the experience of a remote backend without requiring cloud credentials. The patterns you build here transfer directly to production backends such as Amazon S3 with DynamoDB locking, or Terraform Cloud.

## Learning outcomes

By the end of this lab you will be able to:

- Configure a remote state backend and explain what it adds over local state.
- Separate environments so that `dev` and `prod` keep independent state.
- Observe state locking and detect and remediate configuration drift.

## Before you start

- Complete lab 4.2 first. You reuse its resource definitions here.
- Work in `~/devops-week4`.
- A local state backend script is provided at [`starter/scripts/start-state-backend.sh`](starter/scripts/start-state-backend.sh). It starts a small HTTP server that stores Terraform state with locking and versioning.

## Steps

### Step 1 — Start the local state backend

```bash
cd ~/devops-week4
./scripts/start-state-backend.sh
```

This starts an HTTP server on `localhost:9090` that stores Terraform state with locking and versioning. Leave this terminal open; the backend must keep running for the rest of the lab. Run the remaining steps in a second terminal.

### Step 2 — Set up a directory per environment

```bash
mkdir -p envs/dev envs/prod modules/server
```

Move the resource definitions from your lab 4.2 `main.tf` into the module:

- `modules/server/main.tf` (the `random_pet` and `local_file` resources)
- `modules/server/variables.tf` (a `length` variable for the pet name)
- `modules/server/outputs.tf` (output the generated name)

Create `envs/dev/main.tf` and `envs/prod/main.tf`, each calling the module with a different `length` (`2` for `dev`, `3` for `prod`). In each environment's `terraform` block, add a `backend` block configured to use the local HTTP backend, with a backend key matching the environment name so that `dev` and `prod` write to distinct state paths. A ready-to-copy `backend "http"` block is in the header comment of `scripts/start-state-backend.sh`; use `state/dev` in the dev environment and `state/prod` in prod, and keep the `lock_address` and `unlock_address` lines, since without them Terraform does no locking and Step 4 will not work.

### Step 3 — Initialise and apply each environment

```bash
cd envs/dev && terraform init && terraform apply
cd ../prod && terraform init && terraform apply
```

Inspect both environments. They should produce different pet names and different `hello.txt` contents. On the state backend, confirm that two separate state files now exist, one per environment.

### Step 4 — Demonstrate locking

- In one terminal, run `cd envs/prod && terraform apply` but do **not** type `yes` when prompted.
- In a second terminal, run `cd envs/prod && terraform apply`. Observe that Terraform reports the state is locked and refuses to proceed.
- In the first terminal, type `no` to cancel, releasing the lock.
- Re-run the second terminal's command; it now succeeds.

Locking is what stops two people corrupting shared state by applying at the same time.

### Step 5 — Introduce drift and detect it

Manually delete the managed file, simulating a change made outside Terraform:

```bash
rm envs/prod/hello.txt
terraform plan       # in envs/prod: Terraform detects the drift and plans to recreate the file
terraform apply      # the file is restored
```

Now edit `hello.txt` in `envs/prod` to add an extra line, then `terraform plan` again. Terraform detects the content drift. Run `terraform apply` to restore the desired content. Drift detection is how Terraform keeps reality matching your configuration.

## Success criteria

You have completed this lab when your two environments have separate state files, your second concurrent apply is locked out and your manual changes are detected and remediated by `terraform apply`.

## Stretch goal (optional)

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `start-state-backend.sh: permission denied` | Script is not executable | `chmod +x scripts/start-state-backend.sh` then re-run |
| `terraform init` cannot reach the backend | The backend terminal was closed | Restart the backend script and keep that terminal open |
| Both environments share one state file | Backend keys are identical | Give each environment a distinct backend key matching its name |
| Locking never triggers | The backend is not enforcing locks, or you applied in the same terminal | Use two separate terminals and confirm the backend reports a lock |
