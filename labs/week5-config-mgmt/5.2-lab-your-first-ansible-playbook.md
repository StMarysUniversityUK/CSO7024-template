# 5.2 Lab: Your First Ansible Playbook

> **Week 5 · Lesson 2 · Indicative time: 45 minutes · Supports WLO2**
> Part of the CSO7024 DevOps lab series. Companion to the Week 5 module content.

## Overview

In this hands-on lab you write, run and refine your first Ansible playbook. The lab uses a containerised target on your own computer (an Ubuntu container with Secure Shell (SSH) enabled) so you can practise SSH-style execution without needing remote infrastructure of your own. The playbook installs and configures a small static web service. By the end you will have walked through the full edit-run-verify cycle that is the core operational rhythm of working with Ansible.

## Learning outcomes

By the end of this lab you will be able to:

- Write an inventory and a playbook, and run it against a target over SSH.
- Explain and demonstrate idempotency by re-running a playbook with no effect.
- Use a handler so that an action runs only when something has actually changed.

## Before you start

- Install Docker and Ansible 2.16 or later. See [`SETUP.md`](../../SETUP.md), sections 5 and 7.
- A lab container start script is provided at [`starter/scripts/start-lab-container.sh`](starter/scripts/start-lab-container.sh). It launches the Ubuntu target container and prints its address.
- Work in `~/devops-week5/ansible-basics`.

## Steps

### Step 1 — Verify the lab environment

```bash
ansible --version     # should report Ansible 2.16 or later
docker ps             # the lab container should be running; if not, run scripts/start-lab-container.sh
```

The start script prints the container's IP address and the SSH user to use in your inventory.

### Step 2 — Set up the working directory

```bash
mkdir -p ~/devops-week5/ansible-basics && cd ~/devops-week5/ansible-basics
```

### Step 3 — Write a minimal inventory

Create `inventory.ini`, substituting the container IP and SSH user printed by the start script:

```ini
[web_servers]
lab-target ansible_host=<container-ip> ansible_user=labuser ansible_ssh_private_key_file=~/.ssh/lab_key
```

Test connectivity:

```bash
ansible -i inventory.ini web_servers -m ping
```

You should see a `SUCCESS` / `pong` response.

### Step 4 — Write your first playbook

Create `site.yml`:

```yaml
---
- name: Configure web servers
  hosts: web_servers
  become: true
  tasks:
    - name: Install nginx
      ansible.builtin.apt:
        name: nginx
        state: present
        update_cache: true

    - name: Ensure nginx is started and enabled
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true

    - name: Deploy custom index page
      ansible.builtin.copy:
        src: index.html
        dest: /var/www/html/index.html
        owner: www-data
        group: www-data
        mode: '0644'
```

Create a small `index.html` next to the playbook with any greeting you like.

### Step 5 — Run in check mode first

```bash
ansible-playbook -i inventory.ini site.yml --check --diff
```

Read the output. Tasks marked `changed` (yellow) would change something; tasks marked `ok` (green) would do nothing.

### Step 6 — Apply for real

```bash
ansible-playbook -i inventory.ini site.yml
curl http://<container-ip>/      # should return your custom HTML
```

### Step 7 — Demonstrate idempotency

Run the playbook a second time with no changes. Every task should report `ok` rather than `changed`: Ansible has detected that the desired state is already present and made no changes. This is the central idea of configuration management.

### Step 8 — Add a handler

Modify `site.yml` so that the copy task notifies a handler via `notify: Restart nginx`, and add a `handlers:` block with a `Restart nginx` handler that uses the `service` module with `state: restarted`. Edit the `index.html` greeting, then re-run the playbook. Observe the copy task reporting `changed` and the handler running once at the end of the play. A handler runs only when the thing it watches actually changes.

### Step 9 — Capture a screenshot

Take a screenshot of the second-run output from Step 7 showing every task `ok`. This demonstrates idempotency and is one of the most useful things to keep in your own notes.

## Success criteria

You have completed this lab when your second playbook run reports every task as `ok` rather than `changed`, your handler runs only when the `index.html` content has actually changed and `curl` returns your custom HTML.

## Stretch goal (optional)

Refactor the playbook to use a Jinja2 template for `index.html` with a variable for the site title, and move the nginx configuration into a role. This is closer to how production playbooks are structured. A role stub is provided at [`starter/ansible/roles/web/`](starter/ansible/roles/web/).

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `ping` fails with an SSH error | Wrong IP, user or key path in the inventory | Re-check the values printed by the start script; confirm the key file exists |
| `apt` task fails with a lock error | Another package operation is running in the container | Wait a moment and re-run; the container may still be starting |
| Every run reports `changed` for the copy task | The source file changes each run, or permissions differ | Keep `index.html` stable between runs; set `owner`, `group` and `mode` explicitly |
| Handler never runs | The task it watches reports `ok` | Change `index.html` so the copy task reports `changed` |
