# 5.3 Lab: An Integrated Provision-then-Configure Pipeline

> **Week 5 · Lesson 3 · Indicative time: 45 minutes · Supports WLO3**
> Part of the CSO7024 DevOps lab series. Companion to the Week 5 module content.

## Overview

In this hands-on lab you build an integrated workflow that ties together everything you have learned across Weeks 3, 4 and 5. A Terraform configuration creates a small set of containerised "hosts" using the Docker provider; an Ansible playbook installs nginx on each of them with secrets supplied via Ansible Vault; a Continuous Integration (CI) workflow on GitHub orchestrates the whole sequence. This is the most ambitious lab of the module so far.

## Learning outcomes

By the end of this lab you will be able to:

- Generate an Ansible inventory from Terraform outputs so that provisioning and configuration join up.
- Encrypt and consume secrets with Ansible Vault.
- Orchestrate a provision-then-configure pipeline in CI and supply a vault password from a CI secret.

## Steps

### Step 1 — Set up your working copy and inspect the layout

Assemble your working copy by copying the starter's **contents** (not the `starter/` folder itself) into `~/devops-week5/integrated-lab`, so that `terraform/`, `ansible/`, `tests/` and `.github/workflows/deploy.yml` all sit at its root:

```bash
mkdir -p ~/devops-week5/integrated-lab
cp -r <repo>/labs/week5-config-mgmt/starter/. ~/devops-week5/integrated-lab/
cd ~/devops-week5/integrated-lab
cat README.md
```

Keeping `.github/workflows/deploy.yml` at the root matters: GitHub Actions only runs workflows stored in `.github/workflows/` at the **root** of the repository. The README describes the rest of the directory layout.

### Step 2 — Provision with Terraform

```bash
cd terraform/
terraform init && terraform apply
```

This creates three containerised web hosts using the Docker provider. Notice that Terraform writes its outputs to `../ansible/inventory/terraform_outputs.json`.

### Step 3 — Generate the Ansible inventory from Terraform outputs

```bash
cd ../ansible/
python scripts/generate_inventory.py
```

The script reads the JSON outputs and produces `inventory/inventory.ini`. Verify it lists three hosts. This is how provisioning and configuration stay in step: the configuration step reads what the provisioning step created.

### Step 4 — Encrypt your first secret with Ansible Vault

```bash
ansible-vault create vars/secrets.yml
```

Set a vault password when prompted and remember it. In the editor, add placeholders that stand in for real production secrets:

```yaml
site_admin_email: "admin@example.com"
api_key: "abc123"
```

Save and exit, then confirm the file is encrypted with `cat vars/secrets.yml`.

### Step 5 — Wire the secrets into the playbook

Open `site.yml`. Add `vars_files: [vars/secrets.yml]` at the top of the play. Modify the `index.html` template to include `{{ site_admin_email }}`, and confirm the playbook now requires the vault password to run.

### Step 6 — Run the playbook

```bash
ansible-playbook -i inventory/inventory.ini site.yml --ask-vault-pass --check --diff
```

When satisfied, drop the `--check` flag and run for real. Verify with `curl` that each containerised host responds and that the page contains the templated email address.

### Step 7 — Orchestrate it in CI

Open `.github/workflows/deploy.yml`. It has stages for `terraform plan`, `terraform apply`, `ansible-playbook` and smoke tests, with `TODO` markers. Fill them in, including referencing your vault password from a CI secret named `ANSIBLE_VAULT_PASSWORD`.

In your repository, go to **Settings → Secrets and variables → Actions** and add a secret named `ANSIBLE_VAULT_PASSWORD`, set to your vault password. Push your changes and watch the workflow run.

### Step 8 — Capture screenshots

Take a screenshot of the green CI workflow showing every stage succeeding. This is excellent evidence of an integrated pipeline working end to end.

## Success criteria

You have completed this lab when your CI workflow runs `terraform apply`, generates an inventory, runs Ansible with vault-encrypted secrets and your hosts respond correctly to smoke tests.

## Stretch goal (optional)

Add a smoke-test stage that uses the supplied [`starter/tests/smoke_test.py`](starter/tests/smoke_test.py) to verify all three hosts are responding correctly, and make the workflow fail if any host fails the smoke test.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `generate_inventory.py` finds no hosts | Terraform outputs not written yet | Run `terraform apply` first and confirm `terraform_outputs.json` exists |
| Playbook runs without asking for a vault password | `vars_files` not added, or the file is not encrypted | Re-check Step 5; confirm `vars/secrets.yml` is encrypted |
| CI fails at the Ansible stage with a decryption error | The CI secret is missing or misspelt | Confirm a secret named exactly `ANSIBLE_VAULT_PASSWORD` exists in repository settings |
| Hosts unreachable from CI | The Docker hosts are local to the runner | Run the provision and configure stages on the same runner job, as the workflow template does |
