# 4.2 Lab: Your First Terraform Configuration

> **Week 4 · Lesson 2 · Indicative time: 45 minutes · Supports WLO2**
> Part of the CSO7024 DevOps lab series. Companion to the Week 4 module content.

## Overview

In this hands-on lab you write, plan and apply your first Terraform configuration. The lab uses the `random` and `local` providers, which run entirely on your own computer and do not require cloud credentials. By the end you will have walked through the full plan-apply-change-destroy cycle, which is the core operational rhythm of working with Terraform regardless of which provider you eventually use.

## Learning outcomes

By the end of this lab you will be able to:

- Write a small Terraform configuration in HashiCorp Configuration Language (HCL) using providers and resources.
- Explain the difference between desired state and current state, and how `terraform plan` bridges the two.
- Run the plan-apply-change-destroy cycle and read what Terraform proposes at each step.

## Before you start

- Install Terraform. See [`SETUP.md`](../../SETUP.md), section 6. No cloud account is needed.
- Create a working directory for the week: `mkdir -p ~/devops-week4 && cd ~/devops-week4`.
- A starter configuration is provided in [`starter/`](starter/). You can copy it as a starting point or write the files from scratch as below.

## Steps

### Step 1 — Declare the providers

Create `main.tf` and declare the two local providers so Terraform knows what it is working with:

```hcl
terraform {
  required_providers {
    random = {
      source  = "hashicorp/random"
      version = "~> 3.6"
    }
    local = {
      source  = "hashicorp/local"
      version = "~> 2.5"
    }
  }
}
```

### Step 2 — Initialise the working directory

```bash
terraform init
```

Terraform downloads the providers into a `.terraform` directory and writes a lock file. Read the output: it tells you which provider versions it selected.

### Step 3 — Add two resources

Add a `random_pet` resource that generates a friendly name, and a `local_file` resource that writes that name into `hello.txt`:

```hcl
resource "random_pet" "name" {
  length = 2
}

resource "local_file" "hello" {
  filename = "${path.module}/hello.txt"
  content  = "Hello from ${random_pet.name.id}\n"
}
```

Notice that `local_file` depends on `random_pet`: Terraform works out the order for you from the reference.

### Step 4 — Plan

```bash
terraform plan
```

Read the plan. Terraform should report **two resources to add** and nothing to change or destroy. The plan is the proposal; nothing has happened yet.

### Step 5 — Apply

```bash
terraform apply
```

Type `yes` when prompted. Terraform creates the two resources. Confirm the result:

```bash
cat hello.txt
```

It should contain your generated pet name.

### Step 6 — Change and re-apply

Edit the `length` of `random_pet` from `2` to `3`, then plan and apply again. Read the plan carefully: Terraform proposes to **replace** the pet (its name changes) and **update** the file. This is the change half of the cycle.

### Step 7 — Destroy

```bash
terraform destroy
```

Type `yes`. Terraform removes the file and clears its record of the resources from state.

## Success criteria

You have completed this lab when your `terraform plan` shows two resources to add, your `terraform apply` produces a `hello.txt` file containing your generated pet name and your `terraform destroy` removes the file and clears the state.

## Stretch goal (optional)

If you already have an Amazon Web Services or Microsoft Azure account, repeat the same shape of configuration against a real provider (for example an S3 bucket or a resource group). The plan-apply-change-destroy cycle is identical; only the resource types change. Be careful to destroy afterwards so you incur no cost.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `terraform init` fails to download providers | Network restriction or a typo in the source | Check the `source` strings and your connectivity |
| Plan shows zero resources | The resource blocks were not saved, or are in a different file | Confirm `main.tf` contains both resources and is in the current directory |
| `hello.txt` not created after apply | The apply was cancelled, or `content` references the wrong attribute | Re-run `terraform apply` and answer `yes`; check the `${random_pet.name.id}` reference |
| Apply wants to replace everything unexpectedly | Provider versions changed | Pin versions as shown and re-run `terraform init -upgrade` |
