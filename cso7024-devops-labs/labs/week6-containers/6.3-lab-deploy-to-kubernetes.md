# 6.3 Lab: Deploy to Kubernetes

> **Week 6 ┬Ę Lesson 3 ┬Ę Indicative time: 45 minutes ┬Ę Supports WLO3**
> Part of the CSO7024 DevOps lab series. Companion to the Week 6 module content.

## Overview

In this hands-on lab you deploy the Python application from labs 6.1 and 6.2 to a single-node Kubernetes cluster. You write Deployment and Service manifests, scale the deployment, perform a rolling update and observe the cluster's self-healing behaviour by deliberately killing a pod. This is the most ambitious lab of the module and the most direct preparation for the Kubernetes-based deliverable in Assessment 3.

## Learning outcomes

By the end of this lab you will be able to:

- Write Deployment and Service manifests for a small application.
- Scale a deployment and perform a rolling update and rollback.
- Explain and observe self-healing through the reconciliation loop.

## Before you start

- Complete labs 6.1 and 6.2 first. You deploy the image you built in 6.1.
- Install `kubectl` and a single-node cluster (k3s or minikube). See [`SETUP.md`](../../SETUP.md), section 8.
- Stub manifests are provided at [`starter/k8s/manifests/`](starter/k8s/manifests/): `deployment.yaml`, `service.yaml` and `configmap.yaml`, each with `TODO` comments where you fill in values.
- Work in `~/devops-week6`. `cd ~/devops-week6/k8s-lab`.

## Steps

### Step 1 ŌĆö Verify the cluster

```bash
kubectl version          # reports both client and server versions
kubectl get nodes        # shows a single Ready node
kubectl get namespaces   # includes default, kube-system and others
```

### Step 2 ŌĆö Look at the starter manifests

```bash
cd ~/devops-week6/k8s-lab && ls manifests/
```

The directory contains stub `deployment.yaml`, `service.yaml` and `configmap.yaml` files.

### Step 3 ŌĆö Fill in the Deployment manifest

Open `manifests/deployment.yaml`:

- Set `metadata.name` to `cso7024-app`.
- Under `spec`, set `replicas: 3`.
- Under `spec.template.spec.containers[0]`, set `image: localhost:5000/cso7024-app:0.2` (the image you pushed in lab 6.1), `name: app`, `ports: [{containerPort: 8000}]`.
- Add a `livenessProbe:` and a `readinessProbe:`, both using `httpGet: { path: /health, port: 8000 }`, with sensible initial-delay and period values (20 seconds and 10 seconds for liveness; 5 seconds and 5 seconds for readiness).
- Add a `resources:` block with `requests: { cpu: 100m }` and `limits: { cpu: 250m }`. The CPU request is what the HorizontalPodAutoscaler in the stretch goal uses to compute utilisation; without it the autoscaler reads `<unknown>` and never scales.

### Step 4 ŌĆö Fill in the Service manifest

Open `manifests/service.yaml`:

- Set `metadata.name` to `cso7024-app`.
- Set `spec.type: ClusterIP`.
- Set `spec.selector: { app: cso7024-app }` so the Service finds pods labelled with that key.
- Set `spec.ports[0]: { port: 80, targetPort: 8000 }`.

### Step 5 ŌĆö Apply the manifests

```bash
kubectl apply -f manifests/
kubectl get deployment cso7024-app --watch
```

The replica count climbs from 0 to 3 as pods come up. Then:

```bash
kubectl get pods         # three pods in Running state once readiness probes pass
```

### Step 6 ŌĆö Reach the application

The Service is `ClusterIP`, so it is only reachable from inside the cluster. Expose it locally:

```bash
kubectl port-forward service/cso7024-app 8080:80
```

From a second terminal:

```bash
curl http://localhost:8080/health
```

### Step 7 ŌĆö Scale up and down

```bash
kubectl scale deployment cso7024-app --replicas=5
kubectl get pods         # now five pods
kubectl scale deployment cso7024-app --replicas=3
```

### Step 8 ŌĆö Roll out an update

In your lab 6.1 application directory (`~/devops-week6/docker-lab`), edit `app.py` to change the response message, rebuild the image as `cso7024-app:0.3`, tag and push it to the local registry as `localhost:5000/cso7024-app:0.3`, then update the Deployment:

```bash
kubectl set image deployment/cso7024-app app=localhost:5000/cso7024-app:0.3
kubectl rollout status deployment/cso7024-app
curl http://localhost:8080/
```

The rolling update progresses without dropping traffic.

### Step 9 ŌĆö Roll back

```bash
kubectl rollout undo deployment/cso7024-app
```

Observe the rollout going in reverse and the previous version coming back online. Verify with `curl`. Rollbacks are a routine engineering action, not an emergency.

### Step 10 ŌĆö Test self-healing

```bash
kubectl get pods
kubectl delete pod <pod-name>
kubectl get pods --watch
```

Kubernetes notices the deleted pod, schedules a replacement and brings it to `Running` within seconds. The Service keeps serving traffic the whole time because the other two pods stay healthy.

### Step 11 ŌĆö Capture screenshots

Take a screenshot of `kubectl get pods` showing three healthy pods, the `kubectl rollout status` output showing a successful rolling update and the recovery of a deleted pod, for your ePortfolio.

## Success criteria

You have completed this lab successfully if your three pods come up healthy, scaling moves the replica count up and down, the rolling update completes without dropping traffic and deleting a pod triggers an immediate replacement.

## Stretch goal (optional)

Add a HorizontalPodAutoscaler that scales replicas between 2 and 8 based on average CPU utilisation. If `kubectl top pods` returns figures, you are ready. Use `kubectl run` or a simple load-generation script to drive CPU up and watch the autoscaler add replicas; let CPU drop and watch them scale back down.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Pods stuck in `ImagePullBackOff` | The cluster cannot reach `localhost:5000` | Make sure the local registry is running and reachable from the cluster; on minikube you may need to load the image with `minikube image load` |
| Pods never become Ready | Readiness probe path or port is wrong | Confirm the probe targets `/health` on port 8000 |
| Rolling update replaces all pods at once | A single replica, or surge settings | Use `replicas: 3` so the update can proceed gradually |
| Autoscaler shows `<unknown>` for CPU | No CPU request set, or metrics server absent | Set `requests.cpu` and ensure a metrics server is installed |
