# 6.2 Lab: A Multi-Container Deployment with Docker Compose

> **Week 6 · Lesson 2 · Indicative time: 45 minutes · Supports WLO2**
> Part of the CSO7024 DevOps lab series. Companion to the Week 6 module content.

## Overview

In this hands-on lab you extend the single-container deployment from lab 6.1 into a multi-container deployment using Docker Compose. You run the Python application alongside a Postgres database and an nginx reverse proxy, declare the relationships between them in a `docker-compose.yml`, exercise the deployment as a daily-driver development environment and observe how the three services find each other on the Compose network.

## Learning outcomes

By the end of this lab you will be able to:

- Declare a multi-service application in a `docker-compose.yml`, including dependencies and health checks.
- Explain how services on a Compose network find each other by name.
- Distinguish a named volume from a bind mount and choose the right one for data and for code.

## Steps

### Step 1 — Open the starter files

```bash
cd ~/devops-week6/compose-lab && cat README.md
```

The README describes the layout: a local `app/` (the shared application you copied in, with its Dockerfile from lab 6.1), `db/` (a Postgres init script), `proxy/` (an nginx configuration) and a top-level `docker-compose.yml` with `TODO` markers. The `app` service builds from the local `app/` directory, which keeps the lab self-contained and lets you rebuild in place when you change the code later.

### Step 2 — Inspect the starter Compose file

Open `docker-compose.yml`. Note the three services (`app`, `db`, `proxy`), the placeholder definitions and the `TODO` comments showing where to fill in image names, environment variables, ports, volumes and `depends_on` relationships.

### Step 3 — Fill in the application service

- Set `build: ./app` and `image: cso7024-app:0.2` so Compose builds the application you copied in and tags it as the image from lab 6.1.
- Declare `environment:` keys for `DATABASE_URL` (referencing the `db` service by name), `PORT` (set to `8000`) and `LOG_LEVEL` (set to `info`).
- Declare `depends_on:` with `db: { condition: service_healthy }`.
- Declare a `healthcheck:` that calls `/health` every 10 seconds. The image is based on `python:3.12-slim`, which has no `curl`, so use Python: `test: ["CMD", "python", "-c", "import urllib.request,sys; sys.exit(0) if urllib.request.urlopen('http://localhost:8000/health').status==200 else sys.exit(1)"]`. (The starter already contains this line.)

### Step 4 — Fill in the database service

- Use `image: postgres:16-alpine`.
- Set the standard Postgres environment variables (`POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`), reading the password from the project `.env` file via `${DB_PASSWORD}`. Use `appuser` and `appdb` to match the init script and the proxy configuration.
- Declare a named volume `db-data:` mapped to `/var/lib/postgresql/data`.
- Declare a `healthcheck:` that runs `pg_isready -U appuser -d appdb` every 5 seconds.

### Step 5 — Fill in the reverse-proxy service

- Use `image: nginx:1.27-alpine`.
- Mount the supplied `proxy/nginx.conf` as `/etc/nginx/conf.d/default.conf` via a bind mount.
- Declare `depends_on:` with `app: { condition: service_healthy }`.
- Publish port 80 on the host as `ports: ["80:80"]`.

### Step 6 — Set the .env file

```bash
cp .env.example .env
```

Set `DB_PASSWORD` to a reasonable test value. Confirm that `.env` is listed in `.gitignore`.

### Step 7 — Bring the deployment up

```bash
docker compose up -d
docker compose ps
```

Confirm all three services report `healthy` after a minute or so. If any shows `unhealthy`, investigate with `docker compose logs <service-name>`.

### Step 8 — Exercise the deployment

Reach the application through the nginx reverse proxy, then check the health endpoint:

```bash
curl http://localhost/
curl http://localhost/health     # should return 200
```

Modify the application's response message in `app/app.py`, then rebuild and redeploy:

```bash
docker compose up -d --build
```

Observe that only the `app` service is rebuilt and restarted; `db` and `proxy` are untouched.

### Step 9 — Inspect the network and the volume

```bash
docker network ls
docker network inspect compose-lab_default
```

The three services are attached to that network and find each other by name (`app`, `db`, `proxy`) rather than by IP address. Then check data persistence:

```bash
docker volume ls
docker compose down        # without -v, so the volume survives
docker compose up -d
```

Verify that the data in Postgres persists across the restart.

### Step 10 — Capture screenshots

Take a screenshot of `docker compose ps` showing all three services healthy, and a screenshot of `curl http://localhost/` returning a meaningful response, for your ePortfolio.

## Success criteria

You have completed this lab successfully if `docker compose ps` shows all three services healthy, the nginx reverse proxy routes traffic to the application, the application can read and write to Postgres and the database's data persists across `docker compose down` and `up`.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| App starts before the database is ready and crashes | `depends_on` without a health condition | Use `depends_on: { db: { condition: service_healthy } }` |
| `.env` accidentally committed | It was added before `.gitignore` covered it | Rotate the password, regenerate `.env` and confirm `.gitignore` lists `.env` |
| Database data disappears between runs | A bind mount was used for data | Use a named volume for `/var/lib/postgresql/data` |
| `docker compose down -v` wiped the data | `-v` removes named volumes | Omit `-v` unless you intend to discard the data |
