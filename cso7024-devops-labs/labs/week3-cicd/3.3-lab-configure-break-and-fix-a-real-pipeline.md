# 3.3 Lab: Configure, Break and Fix a Real Pipeline

> **Week 3 · Lesson 3 · Indicative time: 45 minutes · Supports WLO3**
> Part of the CSO7024 DevOps lab series. Companion to the Week 3 module content.

## Overview

In this hands-on lab you extend the pipeline you wrote in Lesson 2 with a packaging stage, deliberately introduce a build failure, read the logs and fix it. The lab simulates the daily reality of working with Continuous Integration and Continuous Delivery (CI/CD): most of the time you spend with pipelines is not writing them but troubleshooting them. By the end of the lab you should be confident reading pipeline logs as a primary diagnostic tool.

## Learning outcomes

By the end of this lab you will be able to:

- Add a packaging stage that produces a distributable artefact from a successful build.
- Configure branch protection so that `main` only accepts changes that pass the pipeline.
- Diagnose a build failure from the pipeline log alone and recover from it.

## Before you start

- Complete lab 3.2 first. This lab extends the same `ci.yml` workflow.
- The starter workflow at [`starter/.github/workflows/ci.yml`](starter/.github/workflows/ci.yml) includes commented `TODO` markers for the packaging stage.

## Steps

### Step 1 — Add a packaging stage

Extend `ci.yml` with a third stage that runs only after lint and test succeed. It should build a wheel (a `.whl` distributable) for the application and upload it as an artefact:

- Add a minimal `pyproject.toml` to the application so it is buildable (a name, a version of `0.1.0` and the package layout).
- In a new `package` job, install `build`, run `python -m build --wheel` and upload the resulting `dist/*.whl` as an artefact.
- Make the `package` job depend on the `test` job with `needs: [test]`.

### Step 2 — Run and inspect the artefact

Commit and push, then open the Actions tab. When the run finishes, confirm that a wheel artefact is attached to the run and that you can download it. This is the first time your pipeline produces something you could deploy.

### Step 3 — Configure branch protection

In your repository settings, under **Branches**, add a branch protection rule for `main` that requires the CI status check to pass before merging. This makes the pipeline a gate rather than a report: a change that fails cannot reach `main`.

### Step 4 — Deliberately break the build and read the log

Introduce a failure that the **packaging** stage will catch rather than the lint or test stage. For example, bump the version string in `pyproject.toml` to an invalid value (such as `0.1.x`) so the build tool rejects it.

- Commit and push on a branch, then open a pull request.
- Watch the pipeline fail. Note where the failure is reported, read the exact error message in the `package` job log and observe how the failing status blocks the pull request.

### Step 5 — Fix it and confirm the gate works

Correct the version string, push again and watch the pipeline go green. Confirm that the pull request now reports a passing check and can be merged. Branch protection has done its job.

### Step 6 — Capture a screenshot

Take a screenshot of the green run showing all three stages, and a screenshot of the blocked pull request from Step 4.

## Success criteria

You have completed this lab when your pipeline produces a wheel artefact, fails when you introduce the version error and you can identify the failure from the log.

## Stretch goal (optional)

Pin every action to a specific released major version (for example `actions/checkout@v4` and `actions/upload-artifact@v4`) rather than `@main`, and add a short comment explaining why. Then check the current major version of each action in its repository: action versions drift, and pinning to a versioned tag is what keeps a pipeline reproducible.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `package` job runs even when tests fail | Missing dependency between jobs | Add `needs: [test]` to the `package` job |
| `python -m build` fails with "no pyproject.toml" | Project metadata missing | Add a minimal `pyproject.toml` as in Step 1 |
| Branch protection rule has no effect | The required check name does not match the job | Use the exact job or workflow name shown on a completed run |
| "the pipeline failed" but no detail | You have not opened the failing job's log | Always open the specific job and quote the exact error line; this is a transferable diagnostic skill |
